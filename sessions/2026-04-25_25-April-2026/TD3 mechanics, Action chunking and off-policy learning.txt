I’ve read through it. It’s a beautifully pragmatic paper. 

The core tension here is a classic robotics trade-off: giant Vision-Language-Action models (VLAs) have great "situational awareness" but are clumsy at the "last millimeter" of contact (like threading a zip tie). On the other hand, traditional online Reinforcement Learning (RL) is fantastic at fine-motor precision, but it usually starts completely blind, having to relearn how the world works from scratch.

The authors solve this by building a bridge between the two. You can picture it as a strict division of labor:
*   The massive, frozen VLA acts as the eyes and provides a strong initial guess for the movement. 
*   A tiny, fast RL policy acts as a fine-motor co-pilot, making sub-millimeter tweaks to that initial guess.

The key architectural trick that makes this possible is the **RL Token**—a heavily compressed summary of the VLA's internal state. 

Before we dig into how the co-pilot actually learns to tweak the actions, let me ask you a structural question: **Why go through the trouble of training an entirely new "RL Token" using an encoder-decoder bottleneck?** Why couldn't the tiny RL co-pilot just look at the VLA's regular internal embeddings directly?

Pi0 RL Token (RLT) System

Implements the RL Token method (Xu et al., 2025) where a compact RL token compresses the VLA's internal embeddings and a lightweight actor-critic uses it to refine the VLA's actions through online real-world RL.
Walkthrough: running RL on Pi0 RLT
Architecture in one paragraph

Training is split across two long-running processes that talk over gRPC. They can run on different hosts but we run both on the workstation.

    Learner — Holds the TD3 actor + clipped double-Q critic, keeps the offline (demo) and online replay buffers, samples mixed batches, takes gradient steps, ships fresh actor weights to the actor, ingests fresh transitions from the actor, and logs to WandB.
    Actor — Connects to the leader+follower arms, loads the frozen Pi0 VLA, runs the rollout loop (collecting episodes, handling teleop overrides, labelling success/fail), streams transitions to the learner, and periodically pulls in updated actor weights.

The two processes are independent. You always start the learner first (so the gRPC server is up), then bring up the actor when you're ready to run the robot.
Step 1 — Start the learner

python -m hw_control.pi0_rlt.learner \
    --checkpoint-dir checkpoints/pi05-build-block-tower-rlt-v1 \
    --config-name pi05_rl_token_build_block_tower \
    --language-instruction "build a block tower where the purple block forms the base of the tower" \
    --demo-datasets "villekuosmanen/rl_build_block_tower_pilot" \
    --demo-cache-path outputs/rlt_demo_cache.pt \
    --max-episodes-per-dataset 40 \
    --max-demo-pretrain-steps 20000 \
    --batch-size 256 \
    --device cuda \
    --host 0.0.0.0 --port 50051 \
    --wandb-project pi0-rlt

The demo buffer (offline data)

The demo buffer is built from one or more existing LeRobot datasets via --demo-datasets. Both kinds work:

    IL datasets (e.g. villekuosmanen/build_block_tower) — every episode is treated as a success (reward = +1).
    RL datasets (e.g. villekuosmanen/rl_build_block_tower_pilot) — per-episode success/failure comes from the dataset's EpisodeOutcomePlugin; only successes contribute reward = +1.

For each episode the learner runs the frozen VLA on the sampled tick frames to extract the RL token + reference action, then pairs them with the dataset's recorded action as the "executed" action. This is slow (it's a full VLA forward pass per tick, per episode), so the result is cached to --demo-cache-path and reused on subsequent startups.

You should only delete the demo cache when one of these changes:

    The task you're training on changes.
    You want to add more / different demo data.
    The frozen VLA checkpoint changes (different --checkpoint-dir / --config-name).

Otherwise, leave the cache file alone — startup will be near-instant.
The online buffer

The online buffer is built up from transitions the actor sends back during training. Delete the saved online buffer (and any previous learner checkpoint) every time you start a fresh training run — otherwise you'll mix stale actor data with the freshly-initialised critic. To resume an interrupted run intentionally, pass --resume.
Demo pre-training

Before any actor data shows up, the learner runs --max-demo-pretrain-steps of TD3 updates on the demo buffer alone. This warm-starts the actor + critic from imitation-quality data instead of random init. Default is 20000 steps; smaller (~500–2000) is fine for sanity checks.

If you have no demo data at all, set --max-demo-pretrain-steps 0 and instead use the actor's --warmup N flag (Step 2) to first roll out N pure-VLA episodes so the critic has something to learn from before online TD3 kicks in.
Offline-only smoke test

Before plugging in the robot, it's worth running the learner in --offline-only mode to confirm the loss curves are sensible:

python -m hw_control.pi0_rlt.learner \
    --checkpoint-dir checkpoints/pi05-build-block-tower-rlt-v1 \

    --language-instruction "build a block tower where the purple block forms the base of the tower" \
    --demo-datasets "villekuosmanen/rl_build_block_tower_pilot" \
    --demo-cache-path outputs/rlt_demo_cache.pt \
    --max-episodes-per-dataset 40 \
    --offline-only \
    --online-steps 5000 \
    --batch-size 128 \
    --device cuda \
    --wandb-project pi0-rlt

learner/critic_loss 	Decreases over the first ~500 steps

Step 2 — Start the actor

Once the learner is up and either pre-training or idle waiting for transitions, switch to the robot host.
Power-on checks

    Make sure the robot is powered on and the power bank has enough charge for the planned session.
    The power bank can be recharged via the cables on top of the crate, but the robot must be powered off while charging.
    Bring up both the leader and follower arms. If the USB connection drops mid-session, just restart the actor — the arms occasionally need a re-handshake.

Launch command

python -m hw_control.pi0_rlt.actor \
    --checkpoint-dir checkpoints/pi05-build-block-tower-rlt-v1 \
    --config-name pi05_rl_token_build_block_tower \
    --language-instruction "build a block tower where the purple block forms the base of the tower" \
    --learner-host <LEARNER_IP> --learner-port 50051 \
    --episode-timeout-s 30 \
    --device cuda

Useful extras:

    --warmup N — first N episodes run the raw VLA reference chunks only (RL actor disabled). Use this when you have no demo data, to seed the online buffer with on-policy VLA rollouts before TD3 takes over.
    --rl-dataset-name <hf_repo_id> --rl-dataset-root <local_path> — record the rollouts to disk as a LeRobot RL dataset (with the EpisodeOutcomePlugin populated from the success/fail labels). Reuse later as a --demo-datasets source.
    --drop-held-object-at-end-of-episode — open the gripper after each episode so the robot drops anything it picked up before resetting.
    --policy-fps 20 --actions-to-execute 20 — control rate and chunk length (see "Update cadence and action chunking" below).

Setting the start position

When the actor finishes loading the model it will print:

[ACTOR] Teleop the robot to the START position, then press Enter.

Use the leader arm to teleop the follower into a sensible starting pose — somewhere over the manipulation area is usually right. Press Enter to capture that pose. After every episode the robot will interpolate back to this exact position before the next episode starts, which makes training much more consistent.
Episode loop and keyboard controls

After the start position is captured, the actor enters its main loop. Episode lifecycle is driven entirely by the keyboard:
Key 	When 	Effect
Enter 	Out of episode 	Begin a new episode
1 	In episode 	End episode, label as SUCCESS (reward = +1)
0 	In episode 	End episode, label as FAIL (reward = 0)
Space 	In episode (policy) 	Take over with the leader arm — switches to TELEOP
Space 	In episode (teleop) 	End teleop and auto-label SUCCESS
0 	In episode (teleop) 	Label the teleop episode as FAIL instead (press before Space)
Tab 	Any time 	Acknowledge a safety violation
Esc 	Any time 	Quit cleanly (saves data, returns to start position, disconnects)

Notes on the loop:

    An episode auto-ends at --episode-timeout-s (default 30s), labelled as TIMEOUT (reward = 0). Timeouts are paused while teleoperating, so taking over the robot doesn't eat into the timeout clock.
    The end-of-episode pause is normal: the actor flattens the chunk transitions, computes optional sub-chunk subsamples (see [the RLT paper] for why), packages them, and ships them to the learner before unlocking the next episode.
    Teleop mode briefly pauses (~2s) when handing back from human to policy so you can clear the workspace; the VLA chunk cache is reset at the same time.

What to watch while training

Keep WandB and the robot itself in your peripheral vision:

    Action deviation, critic loss, Q-value spiking → the actor is reward-hacking. The robot will usually start shaking or making unpredictable movements at the same time. Stop, kill the actor, and either roll back to an earlier checkpoint or re-tune. Don't fight it.
    Slow but steady success-rate climb → things are working. Don't compulsively tweak hyperparams; different tasks do need different settings, but resist mid-run changes. (A blog post explaining the knobs is on the way.)

Ending the session

Press Esc to stop. The actor will:

    Finalise any in-flight episode and flush the transition queue to the learner.
    Save the recorded RL dataset (if --rl-dataset-name was set), including per-episode outcomes.
    Interpolate the follower back to the start position.
    Disconnect the robot and teleop.

Once the actor prints [ACTOR] Shutdown complete, it's safe to power off the robot. The learner can keep running (e.g. to continue offline updates), or you can stop it with Ctrl-C.
Step 3 — Evaluation

Once you have one or more RL checkpoints saved by the learner (external/openpi/checkpoints/rlt_checkpoints/step_*/checkpoint.pt), use eval_rlt.py to score them against the raw VLA baseline on the real robot.
Launch command

python -m hw_control.pi0_rlt.eval_rlt \
    --checkpoint-dir checkpoints/pi05-build-block-tower-rlt-v1 \
    --config-name pi05_rl_token_build_block_tower \
    --rl-checkpoint \
        external/openpi/checkpoints/rlt_checkpoints/step_070000/checkpoint.pt \
        external/openpi/checkpoints/rlt_checkpoints/step_095500/checkpoint.pt \
    --language-instruction "build a block tower where the purple block forms the base of the tower" \
    --num-rollouts 10 \
    --timeout-s 30 \
    --device cuda \
    --delta-actions

Each rollout cycles through the raw VLA followed by every RL checkpoint listed in --rl-checkpoint, so a single trial scores all variants on the same scene. After each variant finishes (or you press 1/0 to label it), you reset the workspace and continue. Pass --skip-base to leave the raw VLA out of the cycle once you're past the early-comparison stage.
Why eval is hard, and how to make it less unfair

The honest caveat: a real-robot eval is never as controlled as the training rollouts. Two things conspire against you:

    The follower starting pose drifts. The actor captures its start position at the beginning of training and interpolates back to it after every episode (see "Setting the start position"). The eval script does the same thing, but the pose you capture for an eval session is almost certainly not the same pose used during training — you're teleoping by hand, eyeballing it. Even a few centimetres of difference in the gripper's starting (x, y, z) shifts the entire distribution of the first few VLA chunks, and the RL actor is most sensitive precisely there.
    Scene resets are imperfect. Block positions, lighting, and object orientations vary between trials. The raw VLA tends to be more robust to this than the RL actor (which has been optimised on a narrower distribution), so out-of-distribution scenes systematically penalise the RL variant.

A few things help:

    Reuse the same start position across training and eval. If you can, capture the start pose once, save the joint values somewhere, and teleop to roughly the same numbers in both sessions. Even a rough match dramatically tightens the comparison.
    Run paired trials. eval_rlt.py already does this — VLA and every RL checkpoint share the same scene per rollout, so per-trial noise cancels out. Don't compare numbers from separate eval sessions.
    Use enough rollouts. 10 is the bare minimum to see a signal; 20–30 is much better. Success rates on small N are dominated by reset noise, not policy quality.
    Don't over-interpret a single eval. A 7/10 vs 5/10 split is well within reset noise. Rerun the eval with a fresh start position before concluding the RL actor is better (or worse).

When you do get a clean signal — paired trials, same start pose, ≥20 rollouts — the comparison is meaningful. Until then, treat eval numbers as directional.

State representation

The TD3 critic conditions on:

    RL token (2048D) — compressed representation of the VLA's final-layer embeddings, extracted via a learned query in the Pi0RL encoder. Encodes images, language, and proprioception into a single vector.
    Proprioception (7D) — current joint positions.

The actor sees the same state plus a longer reference action chunk (reference_action_len × 7, the VLA's proposed action sequence) as a side input, with 50% dropout during training. The reference is not fed to the critic — Q(x, a) is a function of state and action alone.
Action representation

The actor outputs a full action chunk directly (not a residual on top of the VLA reference). It's a Gaussian with a fixed small std (fixed_action_std) over a tanh-squashed mean in normalised action space, of shape (predicted_action_len × 7). The chunk is then unnormalised and dispensed step-by-step to the robot.

The actor loss (Eq. 5 of the RLT paper) is:

L_actor = -Q(x, a) + bc_beta * ||a - a_tilde||² + jerk_lambda * jerk(a)

where a_tilde is the first predicted_action_len steps of the VLA reference chunk. The BC term is what keeps the actor near the VLA's behaviour unless the critic justifies deviating; with bc_beta=0 the actor is free to do whatever maximises Q.

The critic uses standard TD3 machinery: clipped double-Q, target networks updated by Polyak averaging, and target policy smoothing noise (target_policy_noise, clipped by target_noise_clip) added to the next-state actor mean before bootstrapping.
Update cadence and action chunking

The VLA produces action chunks. The actor-critic operates at chunk boundaries — every actions_to_execute steps, the VLA re-runs to get a new RL token and reference action chunk, and the RL actor produces a new refined chunk. Between boundaries, actions from the cached refined chunk are executed at policy_fps.

The chunk duration in seconds is:

chunk_duration = actions_to_execute / policy_fps

This is independent of whatever FPS the offline demo datasets were recorded at. The demo buffer automatically adjusts its tick interval to match:

tick_interval = round(dataset_fps × actions_to_execute / policy_fps)

So a 20 FPS dataset sampled with actions_to_execute=10, policy_fps=20 yields tick_interval=10 frames (0.5s per tick). A 50 FPS dataset with the same config yields tick_interval=25 frames (still 0.5s per tick). The TD3 update always sees the same temporal cadence regardless of source FPS.

Default configuration (20 FPS, 10 actions/chunk):
Parameter 	Value 	Effect
policy_fps 	20 Hz 	Control loop rate
actions_to_execute 	10 	Actions per VLA chunk
Chunk duration 	0.5 s 	= 10 / 20
Transition rate 	2 Hz 	= 1 / 0.5 (one TD3 transition per chunk boundary)
30s episode 	~60 transitions 	= 30 / 0.5
Effective γ per episode 	0.99^60 ≈ 0.55 	Reasonable horizon

"""RLTPolicy -- TD3 policy for the Pi0 RL Token system.

Matches the RL Token paper (Xu et al. 2025):

- TD3-style clipped double-Q critic training with sparse binary rewards.
- Gaussian actor with fixed small std, outputs action chunks DIRECTLY
  (not residuals).
- BC regulariser L = -Q(x, a) + beta * ||a - a_tilde||^2 (Eq. 5 of paper).
- 50% reference action dropout during both actor and critic updates.
- Chunk-level TD backup: the replay buffer is expected to store chunk
  transitions where `reward` is already the chunk-level return and
  `discount_chunk` = gamma ** predicted_action_len is applied to the
  bootstrap.

The actor receives a longer reference action chunk
(reference_action_len * action_dim) and outputs a shorter predicted
action chunk (predicted_action_len * action_dim).
"""

from __future__ import annotations

import copy

import torch
import torch.nn as nn
import torch.nn.functional as F
from torch import Tensor

from hw_control.pi0_rlt.config import RLTConfig
from hw_control.pi0_rlt.networks import RLTActor, RLTCriticEnsemble


class RLTPolicy(nn.Module):
    """TD3 policy for the RL Token system.

    Public API
    ----------
    select_action(rl_token, proprioception, reference_action) -> action_chunk
    compute_loss_critic(...)  -> scalar
    compute_loss_actor(...)   -> scalar
    update_target_networks()

    Notes
    -----
    The critic does NOT take the reference action as input; only the
    actor does. This matches the paper's formulation where Q(x, a) is
    a function of state and action alone.

    Predicted action tensors are FLATTENED chunks of shape
    (B, predicted_action_len * per_step_action_dim).

    Reference action tensors are FLATTENED chunks of shape
    (B, reference_action_len * per_step_action_dim).
    """

    def __init__(self, cfg: RLTConfig):
        super().__init__()
        self.cfg = cfg

        self.actor = RLTActor(cfg.network, delta_actions=cfg.policy_uses_delta_actions)
        self.critic_ensemble = RLTCriticEnsemble(
            cfg.network, num_critics=cfg.num_critics
        )
        self.critic_target = copy.deepcopy(self.critic_ensemble)
        for p in self.critic_target.parameters():
            p.requires_grad = False

        self.predicted_action_len = cfg.network.predicted_action_len
        self.reference_action_len = cfg.network.reference_action_len
        self.per_step_action_dim = cfg.network.action_dim
        self.predicted_action_dim = self.predicted_action_len * self.per_step_action_dim
        self.reference_action_dim = self.reference_action_len * self.per_step_action_dim

        self._chunk_discount = float(cfg.discount) ** self.predicted_action_len

        self.ref_dropout = cfg.network.ref_action_dropout
        self.bc_beta = cfg.bc_beta
        self.jerk_lambda = cfg.jerk_lambda
        self.register_buffer(
            "_jerk_joint_weights",
            torch.tensor(cfg.jerk_joint_weights, dtype=torch.float32),
        )

        # BC per-element weight for the flattened action chunk.
        # The gripper (last action dim at each timestep) gets a separate
        # weight so its BC contribution can be reduced independently.
        bc_mask = torch.ones(self.predicted_action_len, self.per_step_action_dim)
        bc_mask[:, -1] = cfg.bc_gripper_weight
        self.register_buffer("_bc_mask", bc_mask.reshape(-1))  # (predicted_action_dim,)

        self.target_policy_noise = cfg.target_policy_noise
        self.target_noise_clip = cfg.target_noise_clip

    def set_norm_stats(self, norm_stats: dict, *, use_quantiles: bool = False) -> None:
        """Propagate OpenPi normalisation stats to all sub-networks."""
        self.actor.set_norm_stats(norm_stats, use_quantiles=use_quantiles)
        self.critic_ensemble.set_norm_stats(norm_stats, use_quantiles=use_quantiles)
        self.critic_target.set_norm_stats(norm_stats, use_quantiles=use_quantiles)

    # ------------------------------------------------------------------
    # Inference
    # ------------------------------------------------------------------

    @torch.no_grad()
    def select_action(
        self,
        rl_token: Tensor,
        proprioception: Tensor,
        reference_action: Tensor,
        *,
        deterministic: bool = False,
    ) -> Tensor:
        """Return an action chunk in **raw** (unnormalised) space.

        At inference the reference action is always provided (no dropout).
        """
        was_training = self.training
        self.eval()
        try:
            action, mean = self.actor(
                rl_token, proprioception, reference_action, sample=False
            )
            if not deterministic:
                noise = torch.randn_like(mean) * self.actor.action_std
                action = (mean + noise).clamp(-1.0, 1.0)
            return self.actor.unnormalize_action(action, proprioception)
        finally:
            if was_training:
                self.train()

    # ------------------------------------------------------------------
    # Reference-action dropout
    # ------------------------------------------------------------------

    def _apply_ref_dropout(self, reference_action: Tensor) -> Tensor:
        """Zero out reference_action for a random fraction of the batch.

        Shape preserved: dropout is applied per-example, not per-element.
        """
        if self.ref_dropout <= 0.0:
            return reference_action
        b = reference_action.shape[0]
        keep = (
            torch.rand(b, 1, device=reference_action.device)
            >= self.ref_dropout
        ).float()
        return reference_action * keep

    # ------------------------------------------------------------------
    # TD3 critic loss
    # ------------------------------------------------------------------

    def compute_loss_critic(
        self,
        rl_token: Tensor,
        proprioception: Tensor,
        action: Tensor,
        reward: Tensor,
        next_rl_token: Tensor,
        next_proprioception: Tensor,
        next_reference_action: Tensor,
        done: Tensor,
    ) -> Tensor:
        """Clipped double-Q TD loss with chunk-level bootstrap.

        ``action`` is predicted-length (predicted_action_dim).
        ``next_reference_action`` is reference-length (reference_action_dim).
        Both arrive from the replay buffer in raw (unnormalised) space.
        """
        self._validate_predicted_shape(action, "action")
        self._validate_reference_shape(next_reference_action, "next_reference_action")

        # Clamp to [-1, 1] so the critic trains on the same action range
        # that the actor's tanh output can produce at inference.
        action_n = self.actor.normalize_action(action, proprioception).clamp(-1.0, 1.0)

        with torch.no_grad():
            next_ref_input = self._apply_ref_dropout(next_reference_action)

            _, next_mean = self.actor(
                next_rl_token,
                next_proprioception,
                next_ref_input,
                sample=False,
            )

            noise = (
                torch.randn_like(next_mean) * self.target_policy_noise
            ).clamp(-self.target_noise_clip, self.target_noise_clip)
            next_action = (next_mean + noise).clamp(-1.0, 1.0)

            target_qs = self.critic_target(
                next_rl_token, next_proprioception, next_action
            )
            min_q = target_qs.min(dim=0).values  # (B,)

            td_target = reward + (1.0 - done) * self._chunk_discount * min_q
            if self.cfg.max_q is not None:
                td_target = td_target.clamp(-self.cfg.max_q, self.cfg.max_q)

        q_preds = self.critic_ensemble(rl_token, proprioception, action_n)  # (N, B)
        td_target_expanded = td_target.unsqueeze(0).expand_as(q_preds)
        td_loss = F.mse_loss(q_preds, td_target_expanded, reduction="none")
        return td_loss.mean(dim=1).mean()

    # ------------------------------------------------------------------
    # Jerk regularisation
    # ------------------------------------------------------------------

    def _jerk_penalty(self, action: Tensor) -> Tensor:
        """Per-joint-weighted second-difference penalty over the action chunk.

        Reshapes flat action (B, predicted_action_len * action_dim) to
        (B, T, D), computes the squared second difference per joint per
        timestep, scales by ``_jerk_joint_weights`` (D,), and sums over
        timesteps and joints. Returns shape (B,).

        Returns zeros when the chunk has fewer than 3 timesteps.
        """
        if self.predicted_action_len < 3:
            return action.new_zeros(action.shape[0])
        B = action.shape[0]
        a = action.view(B, self.predicted_action_len, self.per_step_action_dim)
        second_diff = a[:, 2:] - 2 * a[:, 1:-1] + a[:, :-2]  # (B, T-2, D)
        weighted = (second_diff ** 2) * self._jerk_joint_weights  # (B, T-2, D)
        return weighted.sum(dim=(-2, -1))  # (B,)

    # ------------------------------------------------------------------
    # Actor loss
    #   L = -Q(x,a) + β·||a - ã||² + λ_jerk·second_diff(a)
    # ------------------------------------------------------------------

    def compute_loss_actor(
        self,
        rl_token: Tensor,
        proprioception: Tensor,
        reference_action: Tensor,
    ) -> Tensor:
        """Actor loss with Q-maximisation, BC regularisation, and jerk penalty.

        ``reference_action`` is reference-length from the replay buffer.
        The BC penalty compares the actor's predicted-length output against
        the first ``predicted_action_len`` steps of the reference.
        """
        self._validate_reference_shape(reference_action, "reference_action")

        ref_input = self._apply_ref_dropout(reference_action)

        action, _mean = self.actor(
            rl_token, proprioception, ref_input, sample=True
        )

        q_preds = self.critic_ensemble(rl_token, proprioception, action)  # (N, B)
        min_q = q_preds.min(dim=0).values  # (B,)

        # Truncate reference to predicted length for BC comparison.
        # Clamp to [-1, 1]: the actor's tanh output cannot reach values
        # outside this range, so unclamped targets create an irreducible
        # BC loss floor from tail quantile-normalized reference actions.
        ref_truncated = reference_action[:, :self.predicted_action_dim]
        ref_n = self.actor.normalize_action(ref_truncated, proprioception).clamp(-1.0, 1.0)
        bc_penalty = (((action - ref_n) ** 2) * self._bc_mask).sum(dim=-1)  # (B,)

        loss = -min_q + self.bc_beta * bc_penalty
        if self.jerk_lambda > 0.0:
            loss = loss + self.jerk_lambda * self._jerk_penalty(action)

        return loss.mean()

    # ------------------------------------------------------------------
    # Target network EMA
    # ------------------------------------------------------------------

    @torch.no_grad()
    def update_target_networks(self) -> None:
        tau = self.cfg.critic_target_update_weight
        for tp, p in zip(
            self.critic_target.parameters(),
            self.critic_ensemble.parameters(),
            strict=True,
        ):
            tp.data.lerp_(p.data, tau)

    # ------------------------------------------------------------------
    # Serialisation helpers
    # ------------------------------------------------------------------

    def actor_state_bytes(self) -> dict[str, Tensor]:
        """Return CPU actor state dict for streaming to the actor process."""
        return {k: v.cpu() for k, v in self.actor.state_dict().items()}

TD3 is a dense acronym (Twin Delayed Deep Deterministic Policy Gradient), but the underlying mechanism is surprisingly intuitive once you strip away the jargon. At its core, it is an **Actor-Critic** architecture. 

Think of it as a student and a grader:
*   **The Actor (The Co-pilot):** Looks at the state and proposes a physical action chunk.
*   **The Critic (The Grader):** Looks at the state *and* the proposed action, and outputs a "Q-value"—a single number estimating "how much total future reward will this action get us?"

This code cleanly separates how these two learn. Let's look at the structure:

**1. The "Twin" Critics keep the system grounded**
In `compute_loss_critic`, notice this line:
`min_q = target_qs.min(dim=0).values`
Neural networks naturally tend to overestimate how good an action is. TD3 uses *two* parallel critics (a "critic ensemble"). They both score the action, and the algorithm strictly takes the **minimum** score. This prevents the actor from discovering "mirage" actions that fool a single critic into giving a high score but would actually fail in the real world.

**2. The Actor is pulled in two directions**
Look at the very end of `compute_loss_actor`:
`loss = -min_q + self.bc_beta * bc_penalty`
The actor wants to maximize the critic's score (hence minimizing `-min_q`). But it is heavily regularized by `bc_penalty`, which mathematically penalizes the actor for diverging from the frozen VLA's reference action (`a_tilde`).

**3. Why the RL Token matters here**
This entire TD3 training loop happens constantly, with the critic scoring thousands of transitions sampled from the replay buffer. If the input state wasn't a heavily compressed 2048D "RL Token", but rather the raw, tens-of-thousands of high-dimensional embeddings from the giant VLA, computing these Q-values quickly enough for real-time robot control at 20Hz would be computationally crushing.

Grant

Let's pull that equation completely apart. It is arguably the single most important line of code in reinforcement learning.

`td_target = reward + (1.0 - done) * self._chunk_discount * min_q`

To understand the discount part, we first have to ask: **Why do we need a discount factor at all?**

Imagine we removed it entirely, so the equation was just: `td_target = reward + min_q`.

Think about the zip-tie task. The robot gets a `+1` reward at the very end, and `0` for every step before that. 
If we just pass that `+1` backward without changing it, then the Q-value for being 1 step away from finishing is `1.0`. 
The Q-value for being 10 steps away is `1.0`. 
The Q-value for the very start of the episode is `1.0`. 

If every single state on the path to success has a Q-value of `1.0`, what does the Critic's "landscape" look like to the Actor?

It looks perfectly flat. The Actor wants to maximize its Q-value, but if taking a step forward keeps the Q-value at `1.0`, and standing completely still keeps the Q-value at `1.0`, it has no mathematical reason to actually finish the task. It will just dawdle forever.

**This is what the discount factor solves.**

Normally, the discount factor (let's call it $\gamma$, usually something like `0.99`) acts as a "time tax." Every single time-step, the future reward shrinks by 1%. 

So if you are 1 step away, the value is $0.99$. 
If you are 2 steps away, the value is $0.99 \times 0.99 = 0.98$.
Now the landscape is a slope! The Actor is sitting at `0.98` and sees that if it moves forward, the value goes up to `0.99`. It climbs the hill toward the goal because doing the task *faster* yields a higher Q-value.

**Now, how does this apply to "chunks"?**

In a traditional RL system, the robot takes exactly 1 step, so you multiply by exactly 1 $\gamma$. 

But remember, this system outputs a *chunk* of actions. Let's say a chunk is 10 steps long. If the Critic evaluates State A, and the Actor outputs a 10-step chunk that lands the robot in State B, time has fast-forwarded by 10 steps.

If we only multiplied by `0.99` once, the math would be treating that entire 10-step sequence as if it only took a single step. The Critic would overestimate the value. 

Instead, we have to apply 10 steps worth of "time tax" all at once. 

That is what `self._chunk_discount` is doing. In the `__init__` function of the code, you'll see it defined as:
`self._chunk_discount = float(cfg.discount) ** self.predicted_action_len`

It is literally taking $\gamma^{10}$. So if the next state's value (`min_q`) is $1.0$, the value of the current state becomes $0.99^{10} \times 1.0 \approx 0.90$.
